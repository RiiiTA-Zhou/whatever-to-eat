# 随便：今天吃什么

不知道今天吃什么？用一个“会记得你最近吃了什么”的 Agent 给你提建议。

## 目标（你能从它得到什么）

- 依据你的当天偏好（外卖/堂食/自己做饭、口味、忌口、预算、时间、难度）给出推荐。
- 保证“近期不要重复”：在满足偏好的前提下，让一段时间内的餐食类别覆盖更广、重复更少。
- 生成清晰可执行的输出：你能直接点外卖/去吃，或拿着食材清单开始做饭。

## 核心功能

1. **外卖推荐（`mode=delivery`）**
  - 推荐候选店铺/菜品（或菜品类型），附上理由与预计耗时/成本（由实现阶段定义口径）。
  - 外卖模式下不以营养均衡作为推荐目标，但会展示营养标签（用于知情选择）。
2. **堂食推荐（`mode=dine_in`）**
  - 推荐附近餐馆/菜品（或菜品类型），附上理由与预计到达/就餐时长（由实现阶段定义口径）。
  - 堂食模式下不以营养均衡作为推荐目标，但会展示营养标签（用于知情选择）。
3. **做饭推荐（`mode=cook`）**
  - 推荐菜谱/做法，附上食材清单与预计用时、替换建议。
  - 做饭模式将营养均衡作为明确目标，优先给出覆盖更合理的搭配。
4. **核心记忆（近期不重复 + 口味偏好）**
  - 记录你近期吃过的内容（日期、类型、菜品/菜系/主食材/营养标签、是否喜欢等）。
  - 在推荐时对“最近吃过的类别/主食材”做降权或过滤，尽量做到“样样不同”。
  - 记住你的口味偏好与忌口：例如偏爱/不喜欢的口味（麻辣/清淡/甜咸）、料理方式（煎/炖/蒸）、以及过敏或禁忌（海鲜/坚果/乳糖等）。
5. **营养信息与营养均衡（按模式区分）**
  - 为候选餐食生成“营养标签”（实现后可由规则 + LLM 混合推断）。
  - 在 `mode=cook` 下，根据标签尽量确保每餐覆盖以下要素（实现时可配置）：
    - 蛋白质（肉/鱼/蛋/豆制品等）
    - 蔬菜或水果（至少一种，优先多样化颜色/种类）
    - 主食（米饭/面/杂粮/薯类等，避免每次过度单一）
    - 适量脂肪与纤维（由配方/菜品结构推断）
  - 在 `mode=delivery` / `mode=dine_in` 下，不把营养均衡作为主推荐目标，仅展示营养标签与简要说明。

## 用户输入（Agent 需要你提供什么）

建议把输入拆成两类：**当日输入**（每次都可能变化）与**长期偏好**（可持续）

### 当日输入（每次触发推荐时）

- `mode`：`delivery`（外卖）/ `dine_in`（堂食）/ `cook`（自己做饭）
- `people_count`（可选）：用餐人数（影响份量与预算口径：如“每人¥xx”换算总价/份数）
- `time`（可选）：用餐时间段（例如：午餐/晚餐/宵夜），用于控制候选与用时
- `budget`（可选）：预算区间（实现阶段定义单位：如“¥20-40/人”）
- `location`（可选）：用于“附近”推荐（可只存你选定的区域/经纬度或地标）
- `constraints`（可选）：额外限制，例如“不吃辣/最近想清淡/少油”
- `rating_last`（可选）：可用于你对上一顿的主观反馈（喜欢/一般/不喜欢）

### 长期偏好（可持续录入，支持逐步学习）

- `liked_flavors`：喜欢的口味（甜/咸/酸/辣/麻/清淡/重口等）
- `disliked_flavors`：不喜欢的口味
- `diet_restrictions`：忌口/过敏（例如海鲜、花生、乳制品等）
- `cooking_skill`（当 `mode=cook` 时）：0-3 档（小白/熟练/高手）或“用时上限”

## 输出格式（Agent 每次推荐应该给出什么）

建议输出为结构化文本，便于你复用或做前端展示（由实现决定最终格式）。

- `main_recommendation`：1 个主推荐（外卖/堂食：店铺 + 菜品或菜品类型；自己做饭：菜谱名 + 关键做法）
- `alternatives`：2-3 个备选（同样给出简要理由）
- `response`：简述推荐原因，至少覆盖：
  - 偏好匹配（口味/忌口/难度/预算）
  - 近期不重复（相对最近记录的“差异点”）
  - 营养要素（对应的营养标签/覆盖情况）

## 记忆与“不重复”规则（实现时建议的口径）

为了让“近期不重复”稳定可控，需要明确规则来源与优先级：

1. **窗口期**
  - 默认使用最近 `N` 天（例如 14 天）作为多样性窗口。
  - 也可以改为“最近 N 次记录”。实现后可在配置里切换。
2. **相似度维度（用于避免重复）**
  - `dish_category`：菜品类别/菜系（例如川湘/日式/中式家常/轻食等）
  - `primary_ingredients`：主食材集合（例如牛/鸡/鱼/豆腐/虾/面条等，按实现定义粒度）
  - `cooking_style`：料理方式（煎/炸/炖/蒸/凉拌等）
3. **策略**
  - 过滤优先：若触犯忌口或明显重复（同类别 + 同主食材），则直接剔除。
  - 降权其次：轻度重复时降低权重，而不完全删除，以避免候选不足。
  - 在 `mode=cook` 下增加覆盖优先：如果窗口期内某些营养要素/类别覆盖不足，则对“缺口项”做额外加分。

## 营养标签与做饭模式营养均衡（v0 建议）

- 给候选餐食打标签：`protein`、`veg`、`carbs`、`fiber`、`fat` 等（粒度可先粗后细）。
- 标签来源：
  - 自己做饭：来自食谱结构（食材推断即可）。
  - 外卖/堂食：来自菜品描述（可用 LLM 推断标签；或先用固定菜品模板库）。
- 打分与展示策略：
  - `mode=cook`：每项要素满足得分；满足越多得分越高。若用户显式提出“清淡/少油/控糖”等约束，则将对应标签引入规则或权重。
  - `mode=delivery` / `mode=dine_in`：营养标签仅用于结果展示与解释，不作为主排序目标。

## Agent 架构（建议落地到代码时的模块）

本项目已经在 `requirements.txt` 引入了 LangChain（如 `langchain`、`langchain-chroma`、`langchain-openai`）。后续实现建议至少包含：

- `RecommendationEngine`：候选生成 + 过滤 + 打分 + 排序
- `HistoryStore`：历史记录（吃过什么）与反馈（喜欢/不喜欢）存储
- `PreferenceStore`：长期偏好存储（口味/忌口/能力等）
- `NutritionTagger`：为候选打营养标签
- `Explainer`：把“打分结果”翻译成用户可读的 `why_this`

## 候选数据来源与实现概要（v0）

为了让推荐“有数据可用”，建议按就餐方式区分候选来源：

1. `mode=delivery` / `mode=dine_in`（外卖/堂食）
  - 需要用户提供 `location`（可先用你手动选定的区域/地标），以及 `time`、`budget`、`people_count`。
  - 通过**大众点评 API**在附近拉取餐厅/菜品候选，并为每个候选生成结构化条目（店铺、菜品名/类别、描述、价格区间、评分等）。
  - 根据 `constraints` 与 `diet_restrictions` 做过滤（例如不吃辣/忌口/过敏），并结合 `people_count` 将预算口径从“总价”换算到“人均”或“推荐组合份量”。
  - 对菜品描述进行营养与口味标签推断（可用 LLM + 固定模板兜底），用于结果展示与解释；排序目标以偏好匹配、约束满足、近期不重复为主。
2. `mode=cook`（自己做饭）
  - 候选菜谱直接从 [CookLikeHOC](https://github.com/Gar-b-age/CookLikeHOC) 这个库中选取（按菜谱类别/主菜类型做初筛）。
  - 使用 `cooking_skill`、`constraints`（忌口/不吃某类食材/不吃辣等）过滤掉不合适的菜谱。
  - 对选中的菜谱输出：食材清单、步骤简述、预计用时，以及针对 `people_count` 的份量/用量换算建议（先用粗粒度系数，后续再细化）。
  - 给菜谱打营养标签，并将其纳入做饭模式的营养均衡目标，同时参与“近期不重复”的历史约束。

## 本地存储建议（v0）

为保证“记忆”稳定且可控，建议使用本地文件存储（例如 SQLite 或 JSON + 索引）。字段建议：

- `history_entries`：
  - `date`
  - `mode`
  - `meal_id` 或 `dish_name`
  - `dish_category`
  - `primary_ingredients`
  - `nutrition_tags`
  - `user_feedback`（可选）
- `preferences`：
  - `liked_flavors`
  - `disliked_flavors`
  - `diet_restrictions`
  - `cooking_skill`

## 开发与运行（等你实现后补齐命令行/服务接口）

当前仓库尚未提供具体入口代码，但建议的开发步骤：

1. 环境：`pip install -r requirements.txt`
2. 配置：设置 `OPENAI_API_KEY`（如使用 `langchain-openai`）
3. 实现一个最小可用入口：
  - CLI 或一个简易脚本：输入当天 `mode` + 偏好约束，输出推荐结构化结果。
4. 再逐步加入：真实“附近”候选获取、食谱数据库、以及更精细的营养与多样性策略。

## 示例（期望的交互效果）

### 例 1：外卖

- 你输入：`mode=delivery`，偏爱清淡、少油，预算 ¥30-50
- Agent 输出：
  - 主推荐：xxx（为什么适合你：清淡口味 + 近期没吃过同主食材 + 营养标签覆盖蛋白/蔬菜/主食）
  - 备选：yyy / zzz
  - 动作：点单建议（少油/微辣/加青菜等）

### 例 2：自己做饭

- 你输入：`mode=cook`，用时上限 20 分钟，小白，忌口花生
- Agent 输出：
  - 主推荐：xxx（食材清单、预计用时、步骤简述）
  - 替换：没有鸡肉换成豆腐/鸡蛋的替代建议
  - 说明：营养要素覆盖情况与近期不重复理由
